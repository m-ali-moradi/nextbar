# NextBar — Domain Storytelling

This document captures the core business flows of the NextBar system using **Domain Storytelling**. It describes the sequence of activities that occur within and across the five Bounded Contexts to deliver the event management experience.

---

## 1. Bar Operations Context

### **Story 1: Create Bar and initialize stock**
**Actors:** Bartender, System, Event planner
1. **The system** checks whether the **event planner** has defined any bar and stock.
2. If **an initial bar plan exists**:
    - **The system** retrieves the predefined bar details and its stock.
    - **The system** registers the incoming bars and their stock details in the local DB.
3. If **no initial bar plan exists**:
    - **The system** adds locally predefined bars, and stocks.
4. **Bartender** also can create new bar and stock.

### **Story 2: Serve drink**
**Actors:** Bartender, System
1. **A bartender** serves a drink to a customer.
2. **The system** decreases the corresponding product quantity from the bar’s stock.
3. **The system** logs the served drink in the usage log.
4. **The system** updates the log with the timestamp, product, and quantity.
5. **The system** calculates and displays the **total served quantity per drink type**.

### **Story 3: Auto supply request**
**Actors:** System, Bartender
1. After stock is updated, **the system** checks if the quantity for any product is **less than or equal to 5**.
2. If true:
    - **The system** creates a **supply request** for that product.
    - **The system** displays a toast with timer of 10 seconds to the **bartender**, allowing them to **cancel** or **change the quantity** of the request.
    - The request is marked with status **REQUESTED**.
    - After the request is created and if it is still in **REQUESTED** status, **the bartender** can cancel the request.

### **Story 4: Supply request process**
**Actors:** Warehouse Staff, System
1. **Warehouse staff** views all **REQUESTED** supply requests.
2. **Warehouse staff** updates the request status to **IN_PROGRESS** once fulfillment begins.
3. After delivery:
    - **Warehouse staff** marks the request as **DELIVERED**.
    - **The system** adds the delivered product quantities to the bar’s stock.

## 2. Event Planning Context

### **Story 1: Create Event and allocate resources**
**Actors:** Event Planner, System
1. **An event planner** creates a new event with core details (name, date, location).
2. **The system** initializes the event with status **SCHEDULED**.
3. **The event planner** defines logical bars and assigns a planned stock threshold for each.
4. **The event planner** defines logical drop points for empty bottle collection.

### **Story 2: Start Event and bootstrap infrastructure**
**Actors:** Event Planner, System
1. **An event planner** starts the event to mark it as **RUNNING**.
2. **The system** publishes bootstrap events containing all pre-configured bars, drop points, and stock levels.
3. **The system** asynchronously registers these configurations in the downstream Bar and Drop Point contexts.

### **Story 3: Assign Staff to resources**
**Actors:** Event Planner, System
1. **An event planner** assigns staff members (bartenders, warehouse assistants) to specific event resources.
2. **The system** synchronously queries the Identity & Access Service to fetch user details.
3. **The system** associates the specific staff members with the selected bar or drop point.

## 3. Warehouse Context

### **Story 1: Fulfill Bar supply requests**
**Actors:** Warehouse Staff, System
1. **The system** receives an asynchronous supply request generated by a bar.
2. **A warehouse staff** views the dashboard of all **REQUESTED** supply requests.
3. **Warehouse staff** begins fulfillment and updates the request status to **IN_PROGRESS**.
4. **The system** reserves the requested quantity in the warehouse beverage stock.
5. After physical delivery to the bar, **Warehouse staff** marks the request as **DELIVERED**.
6. **The system** consumes the reserved stock and publishes a status update to replenish the bar's local stock.

### **Story 2: Handle Empty Bottle Collections**
**Actors:** Warehouse Staff, System
1. **The system** receives an automated notification that a drop point is full.
2. **The system** registers a new collection task with status **PENDING**.
3. **A warehouse staff** accepts the collection task and sets it to **ACCEPTED**.
4. **Warehouse staff** physically collects the empty bottles and marks the collection as **COLLECTED**.
5. **The system** aggregates the exact count of collected bottles into the total empty bottle inventory.
6. **The system** publishes a lifecycle event to reset the drop point status back to **EMPTY**.

## 4. Drop Point Context

### **Story 1: Receive empty bottles**
**Actors:** Visitor, System
1. **A visitor** returns an empty bottle to a physical drop point.
2. **The system** increments the current empties stock counter.
3. **The system** checks if the current stock meets or exceeds the maximum capacity threshold.

### **Story 2: Notify Warehouse of full capacity**
**Actors:** System
1. If the current empties stock reaches capacity, **the system** sets the status to **FULL**.
2. **The system** automatically changes the state to **NOTIFIED**.
3. **The system** publishes an asynchronous drop point collection event to alert the Warehouse.

### **Story 3: Acknowledge and Reset collection**
**Actors:** System
1. **The system** receives an acceptance confirmation (**ACCEPTED**) from the Warehouse indicating collection is in progress.
2. After physical collection is completed by warehouse staff, **the system** receives a final lifecycle update.
3. **The system** resets the current empties stock to 0 and returns its status to **EMPTY**.

---

## 5. Identity & Access Context (Users Service)

### **Story 1: Register and setup User**
**Actors:** Administrator, System
1. **An administrator** registers a new user with their personal details.
2. **An administrator** assigns specific roles (e.g., Bartender, Event Planner, Warehouse Staff) to the user.
3. **The system** maps these role assignments to specific context services (e.g., Bar Service, Warehouse Service).

### **Story 2: Authenticate User**
**Actors:** User, System
1. **A user** logs into the application using their credentials.
2. **The system** validates the credentials against the user database.
3. **The system** generates a secure JWT containing the user's role assignments and permissions.
4. **The system** tracks the active session using a refresh token to provide seamless continued access.
